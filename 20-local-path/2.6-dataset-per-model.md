# Lesson 2.6: Building a Proper Dataset Per Model Type

Lesson 2.5 gave you the universal rules of a good dataset. This lesson adds the wrinkle that trips people up: **what a good dataset looks like changes with the model family you chose** (Lesson 2.2). A dataset that trains beautifully on one family can underperform on another, because different architectures want to be fed differently. Matching the dataset to the model is the difference between a clean result and a frustrating one.

## Why model type changes the dataset

Different families differ in how they prefer their training material: the resolution they expect, how they handle different image shapes, how many images they want, and how they read captions (the next lesson). Train as if every model wants the same thing and you will fight whichever family does not. So the move is simple: decide your family first, then build the dataset to its preferences.

## The build process, start to finish

The general workflow holds across families, with the specifics filled per model:

1. **Gather or generate source images.** Pull together your raw material, the same bootstrapping idea from Lesson 1.7, now done with more care and at higher quality.
2. **Cull hard.** Remove anything off-model, blurry, duplicated, or awkward. Ruthless culling beats volume every time (Lesson 2.5).
3. **Prepare the images.** Crop, resize, and clean to your chosen family's spec. Consistent preparation, varied content.
4. **Organize.** Lay the files out the way your training tool expects, ready to caption.

## Sourcing your images

Where the images come from shapes everything. You can generate them (often bootstrapped from a strong cloud result you already made), use images you own, or combine both. The non-negotiable rule from Lesson 2.5 still rules here: every image must be unmistakably the same character. Source however you like, but cull to that standard without exception.
